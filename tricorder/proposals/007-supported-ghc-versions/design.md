# Design: Supported GHC Versions

## Status

Implemented. See `README.md` for the work package.

---

## Problem

Tricorder is tested against GHC 9.6, 9.8, 9.10 and 9.12. That set was never
written down as a policy — versions were added when convenient and never
formally removed. There is no documented process for when to add the next major
release or when it is safe to drop an old one.

---

## Approach

### Source of truth for EOL status

[endoflife.date](https://endoflife.date/ghc) tracks GHC release lifecycle and exposes it as a machine-readable API:

```
GET https://endoflife.date/api/v1/products/ghc/
```

Each version entry has an `isEol` boolean. The rule is:

> A GHC version is supported if and only if `isEol` is `false`.

This criterion is externally maintained, unambiguous, and decoupled from any
single vendor's release schedule.

### Single source of version configuration

`nix/package/common.nix` is the canonical list of supported GHC versions:

```nix
let
  default-ghc-version = "9.10.2";
  additional-ghc-versions = [ "9.6" "9.8" "9.12" ];
  ghc-versions = [ default-ghc-version ] ++ additional-ghc-versions;
in {
  inherit default-ghc-version additional-ghc-versions ghc-versions;
  tested-with = map (v: "GHC == ${v}") ghc-versions;
  # ...
}
```

This file is consumed by two places:

**Nix (`flake.nix`).** A `versionToCompilerName` helper converts dot-separated
version strings to haskell.nix attribute names (`"9.10.2"` → `"ghc9102"`), and
`lib.genAttrs` produces one haskell.nix project per version:

```nix
versionToCompilerName = v: "ghc${builtins.replaceStrings ["."] [""] v}";
defaultGhcVersion = versionToCompilerName common.default-ghc-version;
ghcVersions = map versionToCompilerName common.ghc-versions;
```

**Cabal files.** The `tested-with` list generated by `common.nix` populates the
`tested-with:` stanza in each `.cabal` file, keeping Hackage metadata in sync
with the actual test matrix.

Adding or removing a version from the supported set requires editing only
`common.nix` for both of these consumers.

### CI

A reusable composite action at `.github/actions/ghc-versions/action.yml`
derives the matrix from `common.nix` at workflow runtime:

```bash
nix eval --json --file nix/package/common.nix \
  | jq -c '."ghc-versions" | [.[] | "ghc" + gsub("\\."; "")]'
```

This transforms the version strings in `common.nix` into haskell.nix attribute
names (`"9.10.2"` → `"ghc9102"`, `"9.8"` → `"ghc98"`) and outputs them as a
JSON array. A lightweight `ghc-versions` job runs the action and exposes the
result as a job output; the `check` job consumes it via `fromJson`:

```yaml
ghc-version: ${{ fromJson(needs.ghc-versions.outputs.versions) }}
```

Each matrix cell executes:

```
nix build '.#legacyChecks.${{ matrix.system }}.${{ matrix.ghc-version }}.all' -L
```

The same action and pattern is used in `update-materialization.yml`. Adding or
removing a version in `common.nix` now automatically propagates to both
workflows without any further changes.

### Version lifecycle

**Adding a new version.** When a new GHC major version is released:

1. Confirm `isEol` is `false` in the endoflife.date API.
2. Confirm the version (or a patch release of it) is available as a haskell.nix
   attribute in the nixpkgs revision pinned by `flake.lock` (or update the
   pin).
3. Add the version string to `additional-ghc-versions` in `common.nix`, and
   regenerate the cabal `tested-with` stanzas.
4. Fix any build errors the new version surfaces.

CI automatically picks up the new version on the next run — no changes to
workflow files are required.

**Dropping a version.** When `isEol` becomes `true`:

1. Remove the version string from `additional-ghc-versions` in `common.nix`,
   and regenerate the cabal `tested-with` stanzas.
2. No deprecation period. EOL means no upstream security fixes; continued
   testing has no value.

**Checking EOL status.** `https://endoflife.date/api/v1/products/ghc/` (JSON)
or `https://endoflife.date/ghc` (human-readable).

### Default version

`default-ghc-version` in `common.nix` (currently `"9.10.2"`) selects the
version used for `nix develop`, default package builds, and the development
shell. It uses a full patch version string because it is the primary
local-development target and reproducibility matters more there than for the
additional versions. Update it when the current default goes EOL, or when a
newer version has been in the matrix long enough to be trusted as a daily
driver.

## Alternatives Considered

### Support only the current LTS

GHC marks one version as LTS (currently 9.14). Supporting only LTS would shrink
the matrix to two jobs (one per platform), but would leave users on non-LTS
versions without a tested build and without warning. The full non-EOL set costs
CI time, not engineering time.

### Support only actively-maintained versions (`isMaintained = true`)

This would drop 9.6 and 9.8 (both EOAS). Those versions still receive critical security fixes and have active users. Dropping them early creates an unnecessary compatibility cliff; the `isEol` boundary is cleaner and better supported by upstream.

### Automated matrix generation from the endoflife.date API at flake evaluation time

A fetch inside the flake could keep `ghcVersions` current without any manual updates. Rejected: Import From Derivation (IFD) complicates Nix evaluation caching and makes `nix flake check` depend on network availability. The chosen approach (`common.nix` + composite action) automates CI propagation while keeping the authoritative list explicit and reviewable in the repository.

### Keep version strings in `flake.nix` directly

Before this proposal's branch, `ghcVersions` was hardcoded in `flake.nix`. The `common.nix` approach adds the cabal `tested-with` integration as a free benefit, and makes `nix/package/common.nix` a natural home for all package-level metadata (author, license, language, extensions, warnings) that multiple `.cabal` files share.

---

## Trade-offs

**Matrix size.** Five versions × two platforms = 10 CI jobs per push (once 9.6 and 9.14 are added). This is acceptable given that jobs are parallelised and build artifacts are cached via Cachix.

**`common.nix` version strings vs. haskell.nix attribute names.** `common.nix` stores short semver strings (`"9.8"`, `"9.10.2"`) while haskell.nix and the Nix attribute tree use compacted names (`ghc98`, `ghc9102`). The action's `jq` transform bridges the gap. A version string that does not map to a valid haskell.nix attribute (e.g. before a new nixpkgs pin is in place) will produce a matrix cell that fails at `nix build` time rather than at workflow setup time.
